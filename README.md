# sitar — System InformaTion At Runtime

Sitar collects hardware, kernel, network, storage, security, and package
information from a running Linux system and renders it in one or more
structured output formats. It reads `/proc` and `/sys` directly, invokes
standard system tools (`lspci`, `lsblk`, `ip`, `rpm`, `dpkg`, and others)
where direct access is not possible, and assembles the result into a single
report.

Output formats:

- **HTML** — document with table of contents and inline CSS
- **LaTeX** — `scrartcl` class; renders to PDF via `pdflatex` or `xelatex`
- **DocBook** — simplified DocBook XML (article/section structure)
- **JSON** — machine-readable structured output; schema aligned with the
  [Machinery system description format](https://github.com/SUSE/machinery)
- **Markdown** — GitHub Flavoured Markdown
- **all** — produce every active format in one run

Sitar must be run as `root` to read every source it covers.

> The name. Sitar is also an ancient Indian instrument — the acronym came
> first, the resonance came afterwards.

## A Re-implementation via PCD

The original sitar was written in Perl between 1999 and 2006 at SUSE Linux
Products GmbH (≈3700 lines, GPL-2.0-or-later, twenty named contributors).
This repository contains a complete re-implementation produced by
**Post-Coding Development**
([PCD, "Piccadilly"](https://github.com/mge1512/pcd)).

In PCD, the specification is the human deliverable. Implementation code is
generated by an LLM from the specification and a deployment template. The
target language is never declared in the specification — it is chosen by
the template. The same `spec/sitar.md` produced both a Go binary and a
Rust binary in independent translation runs.

### The numbers

| Artefact | Source | Lines | Hand-written | Authors |
|---|---|---:|---|---|
| Original | `background/sitar-in-perl/sitar.pl` | 3 681 | yes | ≈20, 1999–2006 |
| **PCD specification** | `spec/sitar.md` | **3 058** | reviewed | Claude Sonnet 4.5 via reverse-prompt; reviewed and revised by Matthias G. Eckermann |
| Generated Go binary | `code/go/` | 6 268 | no | LLM (Claude Sonnet 4.5) |
| Generated Rust binary | `code/rs/` | 5 978 | no | LLM (Claude Sonnet 4.5) |

Both generated implementations are accompanied by RPM and Debian packaging,
a man page, independent test suites, and a `TRANSLATION_REPORT.md` recording
every translation decision. No implementation code in `code/` was authored
by a human; no code under `code/` was modified after translation.

### What the case study shows

- **The specification is language-neutral.** Neither the Go nor the Rust
  translation receives any language-specific guidance from `spec/sitar.md`.
  Language is resolved at translation time from the deployment template
  (`spec/cli-tool.template.md`), with milestone-specific hints in
  `spec/cli-tool.go.milestones.hints.md` and `spec/cli-tool.rs.milestones.hints.md`.
- **Scaffold-first held at scale.** The specification is structured into
  eight milestones (M0 through M7). M0 — the scaffold — declares every
  file, type, and interface signature. Each subsequent milestone implements
  a coherent group of BEHAVIORs against that scaffold. The scaffold held
  without modification through seven implementation milestones in both
  target languages.
- **The specification is the source of truth.** Where the generated code
  disagreed with the specification, the specification was fixed and the
  code regenerated — never the other way around.

## Repository Layout

```
sitar/
├── README.md
├── spec/                                ← human-authored
│   ├── sitar.md                         ← the PCD specification
│   ├── sitar.implementation.hints.md    ← language-neutral hints
│   ├── sitar.go.implementation.hints.md ← Go-specific hints
│   ├── cli-tool.template.md             ← deployment template
│   ├── cli-tool.go.milestones.hints.md  ← Go milestone hints
│   ├── cli-tool.rs.milestones.hints.md  ← Rust milestone hints
│   ├── prompt.md                        ← translator prompt used
│   └── reverse-prompt.md                ← reverse-engineering prompt
│
├── code/                                ← LLM-generated, do not hand-edit
│   ├── go/                              ← Go implementation + packaging
│   ├── rs/                              ← Rust implementation + packaging
│   └── translation_analysis/            ← cross-translation notes
│
└── background/                          ← reference material
    ├── sitar-in-perl/sitar.pl           ← the 1999–2006 original
    └── machinery/                       ← Machinery JSON schemas
```

The `spec/` and `background/` directories are authored by humans and edited
directly. The `code/` directory is the output of LLM translation runs and
is regenerated from the specification, not edited. To change the generated
behaviour, change `spec/sitar.md` and re-run the translator.

## Building and Running

Each generated implementation carries its own build instructions, packaging,
and runtime documentation:

- **Go:** see [`code/go/README.md`](code/go/README.md). Builds with `make`;
  produces an RPM via `code/go/sitar.spec` and a Debian package via
  `code/go/debian/`.
- **Rust:** see [`code/rs/README.md`](code/rs/README.md). Builds with
  `cargo build --release` or `make`; same packaging coverage.
